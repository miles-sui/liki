# 引擎 × LLM 功能分解与数据流

**核心原则：引擎计算 ground truth，LLM 负责自然语言表达。** 引擎是确定性的、可验证的数学系统；LLM 是表达层，接收引擎输出的结构化数据，用自然语言解释给用户。

LLM 绝不参与计算。ChatGPT 算八字会幻觉——因为我们不做 LLM 计算，所以能做到竞品做不到的事。

---

## 1. 总览数据流

```
用户输入
  │
  ├── 出生信息 ──→ 命理引擎 (bazichart.go) ──→ ChartResult ──→ 免费展示
  │                    │
  │                    ├── 八字排盘 (四柱、十神、纳音、大运)
  │                    ├── 五行分布 (element_count)
  │                    └── 用神判断 (身强身弱 + 喜用)
  │
  ├── 评估答案 ──→ 25types引擎 (engine.go) ──→ Profile ──→ 免费展示
  │                    │
  │                    ├── Identity (类型 + 标签)
  │                    ├── Deviation (d向量)
  │                    ├── Proportion (p向量)
  │                    └── Flow (季节生克)
  │
  └── 用户问题 ("我和TA合不合适？")
                    │
        ┌───────────┴───────────┐
        │  引擎数据 (不变的 ground truth)  │
        │  + 当前问题                      │
        │  + 报告 schema 指令              │
        └───────────┬───────────┘
                    │
              LLM (表达层)
                    │
        ┌───────────┴───────────┐
        │  LLM 生成的回答               │
        │  ├── 自然语言解读             │
        │  └── 数据溯源标注             │
        │      "基于：日主甲木 + 寅月身强 + 用神金"
        └─────────────────────────────┘
```

**关键边界：**
- 引擎输出 = 结构化数据（数字、枚举、表），100% 确定，可复现
- LLM 输出 = 自然语言文本，标注了它引用了引擎的哪些数据
- 用户可以在任何其他平台验算引擎数据，但 LLM 的解读是 25types 独有的

## 2. 引擎层——纯计算，零 I/O

引擎层（`internal/ganzhi/`, `internal/tianwen/`, `internal/25types/`, `internal/mingli/`）是零依赖的纯函数。同一个输入永远得到同一个输出。

### 2.1 命理引擎

**文件：** `internal/mingli/bazi/` (chart, match, fortune), `internal/tianwen/` (solar time)

| 功能 | 函数 | 输入 | 输出 |
|------|------|------|------|
| 八字排盘 | `ComputeChart()` | 公历日期时间 + 地点(经度/时区) + 性别 | `ChartResult` (四柱、十神、纳音、藏干、大运、起运年龄、五行计数) |
| 真太阳时 | `ComputeSolarTime()` | 北京时间 + 经度 | 真太阳时 (用于排时柱) |
| 八字合盘 | `ComputeBond()` | 两人 ChartResult | 五维交叉分析 (柱交互、十神互视、纳音关系、神煞互现、结构比较) |
| 节气计算 | `PrecomputeSolarTerms()` | 年份 | 24节气精确日期时间 (用于排月柱和流月分析) |
| 今日能量 | `GetDayEnergy()` | 公历日期 | 日柱干支 + 五行 + 节气月 + 地支冲煞 (用于每日建议) |

**ChartResult 包含的数据（LLM 可引用）：**
- 年柱/月柱/日柱/时柱：天干 + 地支 + 藏干 + 十神 + 纳音
- 日主天干 + 日主五行
- 五行统计：木/火/土/金/水 各多少个
- 大运表：每步大运的起止年龄、天干地支、十神
- 身强/身弱判断 + 用神/喜神/忌神

### 2.2 25types 引擎

**文件：** `internal/25types/` (engine, identity, bond, concord, flow)

| 功能 | 函数 | 输入 | 输出 |
|------|------|------|------|
| 计算 D 向量 | `ComputeD()` | 30题答案 ([]Answer) | `Deviation` [5]float64 (和为0，表示偏离基准的方向) |
| 计算 P 向量 | `ComputeP()` | 30题答案 | `Proportion` [5]float64 (和为1，五元素占比) |
| 类型判定 | `ClassifyIdentity()` | P向量 | `Identity` (Label, ID如"WM"/"FE", Category) |
| Bond 计算 | `ComputeBond()` | 两人 D向量 | `BondResult` (双方元素升降 + Delta) |
| Concord 判定 | `ComputeConcord()` | 两人 D向量 | "顺"/"逆"/"平" (生克平衡方向) |
| Flow 计算 | `ComputeFlow()` | D向量 + 当前节气 | `FlowResult` (本月生你/克你的元素方向) |
| 年 Flow | `ComputeFlowYearly()` | D向量 + 年份 | 12个月的 Flow 数组 |

**BondResult 包含的数据（LLM 可引用）：**
- A→B 能量方向：哪个元素被生、哪个被克、强弱
- B→A 能量方向：同上，反向
- Delta A：A 在 B 影响下的元素变化量
- Delta B：B 在 A 影响下的元素变化量
- Concord：总体生克方向

### 2.3 命名引擎

**文件：** `internal/mingli/qiming/`

| 功能 | 输入 | 输出 |
|------|------|------|
| 五行缺补分析 | ChartResult.element_count | 缺失元素 + 补益优先级（yongShen + xiShen） |
| 名字生成流水线 | 姓氏 + NamingAnalysis | 候选名字列表（五行补益 → 生肖过滤 → 组合生成 → 三才五格 + 音韵并排 → Top 20） |
| 单名评估 | 姓氏 + 名字 + yongShen + 生肖 | NameEvaluation（WuGe + SanCai + 音韵 + 五行匹配 + 生肖备注） |
| 三才五格 | 姓氏 + 名字 | 天格/人格/地格/外格/总格 数理 + 五行 + 吉凶 |
| 音韵分析 | 字符序列 | 声调序列 + 平仄交替判定 |

**汉字五行数据库：** 从 ben-hua 通用规范汉字表（`gsc_pinyin_with_tone.csv`，8105 字）全量加载。CSV 自带五行标注覆盖 ~5915 字，余下 ~2190 字通过 `radicalToElement` 映射表（105 部首）推断，总覆盖率 95.4%（7734/8105）。`//go:embed` 编译进二进制，`init()` 加载到 `CharByRune`（O(1) 单字查询）和 `CharByElement`（按元素批量取字）。

**生肖宜忌：** `data/zodiac.yaml`，12 地支完整覆盖。`ZodiacFromYearBranch()` 查询，`GenerateCandidates` 中通过 `filterByRadicals` 剔除含忌用部首的字（硬淘汰门）。

## 3. LLM 层——纯表达，不计算

### 3.1 LLM 的输入

每次 LLM 调用，system prompt 包含：

```
[系统指令]
你是 25types 的 AI 解读顾问。你的任务是把引擎计算的结构化数据用自然语言解释给用户。
规则：
1. 只能基于提供的引擎数据进行解读，不得编造数据
2. 每条关键解读标注数据来源
3. 如果用户问的问题超出引擎数据范围，诚实告知
4. 使用温暖、有洞察力的语言，不说教

[25types 系统知识]
(见 3.4 节，~800 tokens 理论压缩版)

[用户引擎数据]
八字：
  日主：甲木
  月令：寅月 (木旺)
  身强/身弱：身强
  用神：金，喜神：土
  五行分布：木3 火1 土1 金1 水1
  大运：28-38岁 庚申 (金旺，用神到位)
  ...

25types：
  类型：WM (Wood-Metal)
  五元素占比：木32% 火15% 土18% 金22% 水13%
  ...

[当前报告 Schema — 按 scene 选择]
(命格解读 / 起名 / 关系 / 择日 / 职业方向 / 通用 / 每日建议 / 每日提问)

[当前问题]
用户：<当前问题，只有一条 message，无对话历史>
```

**关键设计：**
- 引擎数据部分 (八字 + 25types) 每次请求都完整注入，保证 LLM 不会"忘记"
- 没有对话历史——每次 LLM 调用 messages 数组只有一条 user message。用户"修改重问"是通过 PUT 更新同一份报告，不是追加追问
- 报告 schema 指令按 scene 选择，约束 LLM 的输出结构

### 3.2 LLM 的输出

每个回答包含两个层次：

```
┌─────────────────────────────────────────────┐
│  自然语言解读 (LLM 生成)                      │
│                                              │
│  你是甲木日主，生于寅月，木气当令，身强。       │
│  甲木为参天大树，你天生有开创力和领导欲……      │
│                                              │
│  ─────────────────────────────────────────  │
│                                              │
│  数据溯源 (从引擎数据自动标注)                  │
│  基于：日主甲木 + 寅月身强 + 用神金 + 木3火1... │
│  [点击查看完整推导]                            │
└─────────────────────────────────────────────┘
```

溯源标注锚定在引擎数据字段上——不是 LLM 凭空编的，每条来源对应一个具体的引擎数据点。具体生成机制见 6.2 节。

### 3.3 LLM 技术选择

**模型：DeepSeek V4 Pro。** 走 Anthropic Messages API 协议（`base_url=https://api.deepseek.com/anthropic`），直接用 `anthropic-sdk-go`，零封装。中文理解 SOTA，成本约为 Claude Opus 的 1/10。

**关键特性：**

| 特性 | 值 | 影响 |
|------|-----|------|
| Context 窗口 | 1M tokens | 大多数对话不需要压缩 |
| 服务端磁盘缓存 | 自动，cache hit 价格 ~1/100 | 引擎数据(不变前缀)自动命中——不需要手动 `cache_control` |
| 流式输出 | SSE，Anthropic 协议原生 | 打字机效果，前端 `EventSource` |
| 输出上限 | 384K tokens | 足够生成起名方案全文 |

不选 `deepseek-reasoner`——引擎已经算完了，LLM 只需要表达，不需要推理链。

### 3.4 25types 理论注入

LLM 知道八字（干支、五行生克），但**不知道 25types**——这是我们的专有体系。每次调 LLM 时，必须把 25types 的核心理论作为系统知识注入 system prompt，否则 LLM 看到 `"类型: WM, Wood 32%, Metal 22%"` 无法正确解读。

注入内容压缩在 ~800 tokens 以内，覆盖以下概念：

```
[25types 系统知识]

## 五元素人格
25types 将五行映射为人格维度。五个元素的人格角色：

| 编码 | 元素 | 角色 | 核心特质 |
|------|------|------|---------|
| W | Wood(木) | 开创者 Pioneer | 方向性生长——推动新事物从0到1，不屈不挠，需要空间 |
| F | Fire(火) | 发光者 Luminary | 从中心向外辐射——感染他人、传递能量、需要观众 |
| E | Earth(土) | 培育者 Cultivator | 斡旋转化——连接各方、培养人事物、需要节奏和秩序 |
| M | Metal(金) | 规范者 Refiner | 收敛提炼——建立标准、优化系统、需要清晰和精确 |
| R | Water(水) | 蕴蓄者 Reservoir | 向下渗透——深度思考、积蓄待发、需要安静和独处 |

木火为"发"(向外释放能量)，金水为"收"(向内收敛能量)，土居中斡旋。

## 状态向量
每个人的人格由 d 向量(形状)和 p 向量(比例)描述。

- **d 向量**: [W,F,E,M,R]，总和为0。d_i>0=偏盛(优势维度)，d_i<0=偏弱(弱势维度)
- **p 向量**: [W,F,E,M,R]，总和为1。对外展示的比例，仅用于展示

## 生克矩阵
- **S (生)**: W→F→E→M→R→W。生者推动被生者
- **C (克)**: W→E→R→F→M→W。克者约束被克者

生克运算只用偏盛元素(d⁺)，偏弱元素不参与力输出。

## 25种原型类型
用户被归类到25种类型之一，类型由两字母ID表示：
- **纯型(5种)**: W, F, E, M, R — 单一元素主导
- **生序复合(10种)**: WF, FW, FE, EF, EM, ME, MR, RM, RW, WR — 生生关系
- **克序复合(10种)**: WE, EW, FM, MF, ER, RE, MW, WM, RF, FR — 克克关系

首字母=主导，次字母=辅助。如 WM = Wood主导+Metal辅助，开创者-规范者混合。

## Bond (关系动力学)
两人之间的关系通过生克矩阵计算：
- **Delta A**: B对A的元素影响量
- **Delta B**: A对B的元素影响量
- **Concord**: "顺"(生>克,正向流动) / "逆"(克>生,存在张力) / "平"(平衡)

解读Bond时：不只说"顺/逆/平"，要说哪个元素在生哪个、哪个在克哪个，
转化为具体的互动模式和相处建议。

## Flow (季节能量投影)
每个节气月有固定的五行能量方向。用户的d向量与当月能量的生克关系
决定本月哪些元素被增强、哪些被抑制。用于流月运势解读。
```

**注入位置：** system prompt 中 [系统指令] 之后、[用户引擎数据] 之前（见 3.1 节 prompt 骨架）。每次 LLM 调用都注入——DeepSeek 服务端缓存自动检测前缀重复。

**理论来源：** `docs/theory/THEORY.md`（完整推导）、`docs/theory/SPEC.md`（编码规范）、`docs/INDEX.md` 第 4 节（元素编码 + 原型矩阵）。此处是压缩版，仅包含 LLM 解读必需的概念。

**SDK 接入：**
```go
import "github.com/anthropics/anthropic-sdk-go"

client := anthropic.NewClient(
    anthropic.WithBaseURL("https://api.deepseek.com/anthropic"),
    anthropic.WithAPIKey(os.Getenv("DEEPSEEK_API_KEY")),
)
resp, _ := client.Messages().Create(ctx, &anthropic.MessageCreateParams{
    Model:    "deepseek-v4-pro",
    MaxTokens: 4096,
    System:   systemPrompt,   // 引擎数据 + 场景指令
    Messages: history,        // 最近 N 轮对话
    Stream:   true,
})
```

**备选：** `LLMClient` interface 隔离具体实现，可切换 Anthropic 原生/OpenAI。

## 4. 场景分解：引擎做什么 vs LLM 做什么

所有场景统一使用 `POST /api/reports` 端点，通过 `scene` 参数选择报告类型。引擎计算结构化数据（免费预览），LLM 生成完整报告（付费）。一份报告一次生成，无追问，无对话历史。

### 4.1 命格解读 — scene: `mingge`

```
用户: "我的命格怎么样？"

引擎提供:
  - ChartResult (四柱、十神、纳音、藏干、大运)
  - 身强/身弱 + 用神/喜神/忌神
  - 格局分类 (建禄格、正官格、食神格...)
  - 五行统计

LLM 生成:
  - 命格判断 (通俗解释日主 + 月令 + 格局)
  - 五行分析 (各元素对性格/人生的影响)
  - 大运时间线解读 (每步 ≥2 句自然语言)
  - 当前运势 (流年 + 流月)
  - 发展建议 (事业/财运/感情)

报告结构 → 见 8.1 节
```

### 4.2 起名 — scene: `naming`

```
用户: 姓李，宝宝 2026/3/15 14:30 出生，男
params: {surname, birth, gender}

引擎提供:
  - ChartResult → 五行缺补分析 (缺水 → 第一补益水，第二补益金)
  - 汉字五行库筛选 (五行=水的字: 涵、泽、沐、清、澜、鸿...)
  - 三才五格计算 (每个候选字 × 姓氏)
  - 按评分排序 → Top 10 候选名单

LLM 生成:
  - 每个名字的深度解释 (五行作用 + 字源典故 + 寓意 + 音韵)
  - 综合推荐 + 其他风格方向 + 小名建议

付费边界: 引擎数据(缺补方向 + 候选名单)免费, LLM 详解付费。
报告结构 → 见 8.3 节
```

### 4.3 择日 — scene: `almanac`

```
用户: 备选婚期 5/1、5/15、6/8，哪个好？
params: {dates[], event_type}

引擎提供:
  - 每日期日的天干地支 + 五行 + 节气位置
  - 与用户日柱的冲/合/刑/害检查
  - 用神契合度评分
  - 综合排序: 5/15 (85分) > 6/8 (72分) > 5/1 (45分，冲日柱)

LLM 生成:
  - 逐日解读 (日能量 + 匹配逻辑 + 建议时段)
  - 冲克风险醒目标注
  - 综合建议 (首选 + 备选 + 理由)

付费边界: 引擎数据(排序表 + 分数)免费, LLM 逐日详解付费。
报告结构 → 见 8.4 节
```

### 4.4 关系 — scene: `relationship`

```
用户: "我和TA合不合适？"
params: {other_user_id} 或 {other_birth}

引擎提供:
  命理 (需两人出生信息): ComputeBond() → 五维交叉分析
  性格 (需两人完成评估): ComputeBond() → Delta A/B + Concord

LLM 生成:
  命理部分: 二人命盘 + 合盘解读 + 关键时间窗口
  性格部分: 两人类型 + Bond 动力学解读 + 摩擦点 + 相处建议
  综合视角 (两部分都有时): 高层串联，不强行融合

数据可用性决定报告包含哪些部分 → 见 8.2 节矩阵。
报告结构 → 见 8.2 节
```

### 4.5 职业方向 — scene: `career`

```
用户: "我适合学什么专业？"
params: {career_type: "gaokao"/"zhuanxing"}

引擎提供:
  - 25types Identity → career_mapping.yaml → 匹配学科/职业列表
  - (可选) ChartResult.用神 → 行业五行方向

LLM 生成:
  - 方向解读 (为什么这些方向适合你的类型特质)
  - 八字补充 (用神对应的行业方向)
  - 发展建议 (当前阶段 + 长远方向)

付费边界: 引擎数据(匹配学科表)免费, LLM 方向解读付费。
报告结构 → 见 8.5 节
```

### 4.6 通用 — scene: `null` (无预置场景)

```
用户: 自由提问, 如 "我该不该复合？"
params: 无

引擎提供:
  - ChartResult + PersonalityProfile (已有的全部引擎数据)

LLM 生成:
  - 按最小完整性约束生成报告: 结论 → 依据 → 展开(≥3维度) → 建议 → 溯源

```

### 4.7 每日建议 — scene: `daily_suggestion`

## 5. System Prompt 结构

每次生成报告，system prompt 包含角色指令、引擎数据、报告 schema 要求。LLM 不自由发挥——输出必须满足 schema。

### 5.1 通用 Prompt 骨架

```
[角色]
你是 25types 的 AI 解读顾问。你的任务是把引擎计算的结构化数据，
生成一份完整、可独立阅读的分析报告。你是有洞察力的专家，不是聊天机器人。

[核心规则]
1. 只基于提供的 [引擎数据] 进行解读，不得编造任何数据
2. 每条关键判断标注数据来源
3. 报告必须包含：结论 → 推导 → 展开(≥3维度) → 建议 → 溯源
4. 报告是完整交付——读者不需要追问就能获得全部信息
5. 不要假装你会算命——是引擎在算，你在解释
6. 使用温暖、有洞察力的语言，但不要过度恭维
7. 最少 500 字

[25types 系统知识]
{{25types_theory}}      ← 3.4 节的理论压缩版，~800 tokens

[引擎数据]
{{engine_data}}         ← 用户的八字 + 25types 数据，JSON 格式

[报告 Schema]
{{report_schema}}       ← 按场景不同，见 5.2
```

### 5.2 各场景 Schema 指令

**命格解读：**

## 7. API 架构

### 7.1 报告端点

```
POST /api/reports              ← 生成报告 (核心端点, auth required)
  Request:  {question: string, scene?: string, params?: {...}}
  Response: {report: Report, stream: SSE}

GET  /api/reports               ← 列出当前用户的所有报告
GET  /api/reports/{id}          ← 获取单份报告详情
PUT  /api/reports/{id}          ← 重问——更新同一份报告 (auth, 重新生成)
DELETE /api/reports/{id}        ← 删除报告
```

**"重问"的含义：** 用户对报告不满意，修改问题后重新生成。PUT 请求携带新的 `question`，后端重新调 LLM 生成，更新 `content` 和 `updated_at`，返回新报告。不创建新记录——同一份报告的迭代。

**预置场景的参数：**

| 场景 | question (可为空) | params |
|------|-----------------|--------|
| `naming` | "给宝宝起名" | `{surname, birth, gender}` |
| `relationship` | "分析我和TA的关系" | `{other_user_id}` 或 `{other_birth}` |
| `almanac` | "选个结婚日期" | `{dates[], event_type}` |
| `mingge` | "我的命格怎么样" | —（引擎数据已有） |
| `career` | "我适合什么专业" | `{career_type: "gaokao"/"zhuanxing"}` |
| `null` (通用) | 必填 | — |

### 7.2 每日端点

**每日建议（被动推送）：**

```
GET  /api/daily/suggestion        ← 每日建议 (auth, 免费)
  Response: {
    suggestion: "...简短建议...",
    type: "color"/"action"/"mood"/"health",
    sources: [...]
  }
```

系统根据今日干支 + 用户命盘自动生成。每日首次请求时生成并缓存（TTL=次日零点），后续请求直接返回缓存。

**每日提问（从建议触发的互动）：**

```
POST /api/daily/question           ← 每日提问 (auth, 免费, 每日限 3 次)
  Request:  {question: "那今天去谈加薪合适吗？"}
  Response: {answer: "...", sources: [...]}

GET  /api/daily/question/remaining ← 今日剩余次数
  Response: {used: int, remaining: int, limit: 3}
```

用户看到每日建议 → 被触发好奇 → 点"问今天的事"提一个和今天相关的问题。严格限定"今天"范围——超出范围引导到完整报告。

**每日功能与报告的区别：**

| | 完整报告 | 每日建议 | 每日提问 |
|---|---|---|---|
| 触发方式 | 用户主动提问 | 系统自动推送 | 用户从建议触发 |
| 内容范围 | 不限 | 今日能量 × 一个维度 | 和今天有关的简单问题 |
| 输出长度 | ≥500 字，结构化 | ≤150 字 | ≤150 字 |
| 费用 | ¥9.9/份 或月费 | 免费 | 每天 3 次免费 |
| 频次 | 按需 | 每天 1 条 | 每天 3 次，不累积 |
| 目的 | 深度决策 + 付费 | 留存底线 | 互动深度 + 报告转化 |
| 今日能量数据 | 不注入 | 注入 | 注入 + 建议上下文 |

**建议类型每日轮换：** 穿搭颜色 → 行事方向 → 情绪提示 → 健康小贴士 → 循环。前端展示对应的图标和标题。

**转化漏斗：** 建议是钩子（"今天适合财务决策"）→ 提问是互动（"那我去谈加薪？"）→ 当用户的需求超过每日范围时引导报告（"你想了解的不只是今天——要不要生成一份完整的职业方向报告？"）。三级：建议留存 → 提问互动 → 报告付费。

**生成流程：**

```
每日建议 (GET /api/daily/suggestion):
  ├── 1. 鉴权
  ├── 2. 检查今日缓存 → 已缓存直接返回
  ├── 3. 查用户引擎数据 (ChartResult)
  ├── 4. 计算今日能量 (GetDayEnergy(today))
  ├── 5. 确定建议类型 (轮换: color/action/mood/health)
  ├── 6. 组装 system prompt → 调 LLM (max_tokens=200)
  └── 7. 缓存 (TTL=次日零点) + 返回

每日提问 (POST /api/daily/question):
  ├── 1. 鉴权 + 检查今日剩余次数
  ├── 2. 查用户引擎数据 + 今日能量 + 今日建议(上下文)
  ├── 3. 组装 system prompt:
  │      system = 每日提问角色指令 + JSON(引擎数据+今日能量+建议)
  │      messages = [{role: "user", content: question}]
  ├── 4. 调 LLM (max_tokens=200)
  ├── 5. 扣减今日次数 (TTL=次日零点)
  └── 6. 返回: {answer, sources, remaining}
```

### 7.3 报告生成流程

```
POST /api/reports {question, scene, params}
  │
  ├── 1. 鉴权 + 校验付费额度
  ├── 2. 查用户引擎数据 (ChartResult + PersonalityProfile)
  ├── 3. 如果有 scene → 调对应的引擎计算 (naming/dates/relationship)
  ├── 4. 选择报告 schema 指令 (按 scene)
  ├── 5. 组装 system prompt:
  │      system = 角色指令 + JSON(引擎数据) + 报告 schema 指令
  │      messages = [{role: "user", content: question}]
  │      ← 注意：没有对话历史，只有当前这一个问题
  ├── 6. 调 DeepSeek V4 Pro (Anthropic SDK, stream=true)
  ├── 7. SSE 流式返回 Markdown 报告
  └── 8. 保存到 ask_reports 表
       (PUT 重问: 更新已有记录而非新增)
```

### 7.4 付费边界

| 端点 | 免费 | 付费 |
|------|------|------|
| `POST /api/bazi/chart` | 完整八字排盘 (已有) | — |
| `POST /api/reports` (命格) | 命盘概览 (引擎数据) | 完整命格解读报告 |
| `POST /api/reports` (起名) | 五行缺补 (引擎数据) | 完整起名报告 (候选+详解) |
| `POST /api/reports` (关系) | 二人概览 (引擎数据) | 完整关系分析报告 |
| `POST /api/reports` (择日) | 日期排序表 (引擎数据) | 完整择日报告 (逐日详解) |
| `POST /api/reports` (职业) | 个人画像 (引擎数据) | 完整职业方向报告 |
| `POST /api/reports` (通用) | 问题预览 (一个问题 insight) | 完整通用报告 |
| `GET /api/daily/suggestion` | 完整每日建议 | — |
| `POST /api/daily/question` | 完整每日提问 (日限 3 次) | — |

**付费触发：** 引擎数据部分在 POST 请求返回体中标记 `free_preview: true`，完整报告标记 `paid: true`。前端先渲染免费部分 → 用户付费 → 流式渲染 LLM 生成部分。

### 7.5 数据模型

```sql
CREATE TABLE ask_reports (
    id          TEXT PRIMARY KEY,    -- UUID
    user_id     INTEGER NOT NULL,
    question    TEXT NOT NULL,        -- 用户最新提问
    scene       TEXT,                 -- "naming" / "relationship" / "almanac" / "mingge" / "career" / null
    content     TEXT NOT NULL,        -- Markdown 报告全文
    sources     TEXT,                 -- JSON: 引擎数据引用
    params      TEXT,                 -- JSON: 场景参数 {surname, birth, ...}，重问时可复用
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL,        -- 重问时更新
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**一张表，无对话历史。** 每份报告独立完整，可被重写（PUT 更新），不可被追问（无 parent/thread 字段）。

```
  即使问题很窄("我该不该复合？")，报告也必须包含：
  1. 结论 —— 一句话回答
  2. 依据 —— 基于引擎数据的推导

  4. 建议 —— 可执行的下一步
  5. 溯源 —— 标注数据来源
  最少 500 字。报告是完整交付，不是一句话回复。
```

## 11. 关键设计决策

1. **一问一报告，无对话。** 用户提一个问题，系统生成一份完整报告。没有追问、没有多轮对话、没有 session 管理。不满意 = 修改问题重问 → 更新同一份报告。

2. **引擎算完，LLM 填空。** 引擎提供结构化数据（表、列表、分数），LLM 在报告的"解读"部分填入自然语言。Schema 约束 LLM 的输出结构。

3. **报告 = 引擎数据（免费预览）+ LLM 文案（付费）。** 付费线清晰——用户先看到引擎算出来的数据和一句话 insight，付费后获得完整报告。

4. **退化的 Agent——无 tool calling，无对话历史。** 每次 LLM 调用 messages 数组只有一条 user message。引擎数据完整注入 system prompt。

5. **报告是可变版本。** PUT `/api/reports/{id}` 用新的 question 重新生成同一份报告。用户不是在"追问"——是在"说清楚"。

6. **Schema > 自由发挥。** 8 个 prompt 模板定义了输出结构。LLM 在结构内填充，不乱跑。

7. **LLM 答案是"可溯源的"。** 每条回答标注引擎数据来源——用户不是信任 AI，是信任引擎数据。

8. **DeepSeek V4 Pro + Anthropic SDK。** 1M context，服务端自动缓存，中文 SOTA。通过 Anthropic Messages API 调用。
